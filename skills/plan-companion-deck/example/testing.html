V1.1+CLI · QA Module · Stratégie de test 2026-05-09

Tester ce qu'on a livré, en cassant ce qu'on peut casser.

V1.1, V1.1.5 et le CLI sont sur main, déployés sur syndicable-codika-staging. 99 exports, dont une cinquantaine de callables et une douzaine de triggers. Les tests qui ont tourné pendant les phases étaient bornés à l'émulateur ; ce plan attaque la même surface, mais en vrai — Firestore réel, Resend réel, Pub/Sub réel, puppeteer réel. Tout ce que l'émulateur ne sait pas reproduire.

01 · La thèse

Tester sur staging, parce que l'émulateur ment par omission.

On a livré beaucoup en peu de temps : V1.1 (assemblées, convocations, session live, PV), V1.1.5 (vue copropriétaire — magic link self-serve), et un CLI qui mappe chaque Cloud Function. Les tests qui ont tourné pendant ces phases étaient des tests Jest contre l'émulateur Firebase. Ils valident la logique, mais pas le déploiement. Le déploiement ajoute une couche entière qui ne se voit qu'en staging.

Ce que l'émulateur ne reproduit pas : les cold starts de Cloud Functions (et la latence variable des premiers appels après inactivité), le vrai comportement de Pub/Sub côté triggers (timing, ordre, retry), le rendu PDF par puppeteer avec son chromium en cold start, l'envoi réel des e-mails Resend (DNS, SPF, DKIM, format multi-client), les quotas de débit sur les callables publics, les règles Firestore appliquées à chaud sur des UIDs réels.

Le test qu'on veut faire est simple à dire : un agent reproduit le scénario Brugmann sur staging, depuis le CLI, sans ouvrir le navigateur — sauf pour la voter page (qui doit être testée dans un vrai browser). À la fin, le PV existe en PDF, est arrivé dans des inboxes de test, et les chiffres tiennent. Si oui, on est verts. Si non, on diagnostique, on fixe, on redéploie la fonction touchée, on relance.

02 · Périmètre, figé

Ce qu'on teste — et ce qu'on ne teste pas dans cette passe.

Cette passe est de l'integration testing sur staging, pas du load test, pas du pen test, pas de l'audit RGPD. Le périmètre couvre tout ce qui est appelable depuis le CLI, plus la voter page (qui a son propre canal d'auth via magic link).

Inclus · cette passe de QA

  • Chaque callable + trigger atteignable depuis le CLI exercé bout-en-bout sur syndicable-codika-staging
  • Le scénario Brugmann (cli.html §07) tourne proprement du début à la fin, en mode happy path
  • La voter page publique se charge depuis une vraie URL magic-link, le vote est casté, la vue conducteur du syndic se met à jour en moins de 2 s
  • Le PV est généré, exporté en PDF, distribué à un inbox de test owner — vérification de réception et de structure
  • Firestore staging revient à un état propre connu après chaque passe (cleanup CLI ou reset manuel)
  • Les 5 échecs Phase D pré-existants dans assemblies.test.ts sont fixés avant clôture
  • Couverture ~20 cas négatifs documentés (§06) — pour chaque CF, un essai de violation des gardes

Hors · pas dans cette passe

  • Load testing (>500 votants concurrents) — quotient de hardening V1.2
  • Penetration testing du modèle de tokens — audit externe planifié séparément
  • Audit RGPD complet (DPIA, registre des traitements) — chantier juridique distinct
  • Multi-region failover, DR drill — pas avant que syndicable-codika-prod soit cut
  • Tests de compatibilité multi-client e-mail (Outlook, Gmail mobile, Lotus Notes…) — visuel uniquement
  • Régression UI au-delà de la voter page (le conducteur Svelte est testé au sniff seulement)
  • Tests de migration de données IGS (legacy → V1.1) — quand le client lance l'import
03 · Trois axes de test

Trois angles d'attaque qui se complètent.

Une seule de ces approches ne suffirait pas. Le happy path révèle l'intégration ; le negative testing révèle les gardes ; la boucle fix-and-retry est ce qui transforme « j'ai trouvé un bug » en « j'ai expédié un fix ». Les trois tournent en parallèle.

CHOIX 01

End-to-end Brugmann replay.

On scripte une AG complète depuis le CLI : une copropriété, 5 lots avec quotités, 5 propriétaires, 3 résolutions, 2 votants par magic link, clôture, génération + export + distribution du PV. On compare les chiffres finaux à ce qu'on attendait. C'est le smoke test qui tourne à chaque itération — la sentinelle.

La référence pour ce scénario, c'est cli.html §07. On le rejoue à l'identique, sauf qu'on prend une copro scratch sur staging — jamais de touch sur les données IGS réelles. Si ce scénario passe, 80% du périmètre fonctionnel est validé sans avoir cliqué une seule fois.

CHOIX 02

Negative testing par CF.

Chaque Cloud Function a ses gardes : auth (Firebase user ou magic-link token), machine d'états (assembly status, resolution status, attendance state), validation métier (quotités-concerned, quotités>0, expirations). On essaie de violer chacune. Le résultat attendu : un HttpsError typé propre, jamais un 500.

Le tableau §06 liste ~20 scénarios. La règle : si on tombe sur un code d'erreur inattendu (ex. internal au lieu de failed-precondition), ou une stack trace remontée brute, c'est un bug. On le note, on le fixe, on relance.

CHOIX 03

Boucle fix-and-retry.

Quand un test casse : on lit les logs gcloud functions logs, on inspecte l'état Firestore via le CLI, on reproduit en local si besoin. On corrige dans syndicable-app ou syndicable-cli, on rebuild (cd functions && npm run build), on redéploie ciblé (firebase deploy --only functions:<name>), on relance le test.

Les fixes vont directement sur main en commits incrémentaux ; pas de PR par micro-fix. Quand la passe est terminée, on bundle l'ensemble dans une seule PR fix(qa): post-deploy testing sweep pour la trace. Si un fix demande >1h ou touche le contrat d'une CF, il sort de la boucle et passe par une PR dédiée.

04 · La stack de test

Outils auxquels le sub-agent d'exécution a accès.

Pas de framework de test maison, pas d'orchestration custom. On compose la stack à partir d'outils existants : le CLI Syndicable comme harnais principal, firebase et gcloud pour l'inspection, un browser pour la voter page, le dashboard Resend pour vérifier la livraison des e-mails.

  • CLI syndicable Harnais primaire. V1.1+CLI couvre 100% des CFs callables. Profile dédié --profile staging pour pointer sur syndicable-codika-staging.
  • CLI firebase Déploiements ciblés (firebase deploy --only functions:<name> --project syndicable-codika-staging) et inspection Firestore (firebase firestore:data ...) pour des sanity checks rapides.
  • CLI gcloud Lecture des logs CF (gcloud functions logs read <name> --project=syndicable-codika-staging --gen2 --region=europe-west1) — la principale fenêtre sur ce qui s'est vraiment passé en backend quand un test casse.
  • CLI syndicable lot list, owner list, assembly show --watch Inspection Firestore lecture-seule via le CLI — pas besoin d'Admin SDK. Le flag --watch remplace F5 dans le browser pour observer un état évolutif.
  • Web resend.com Dashboard Resend pour vérifier les livraisons d'e-mails (convocation, PV). Inboxes de test : syndicable-test+<scenario>@example.com (plus-addressing) ou comptes Mailtrap si Resend ne le permet pas.
  • Web https://syndicable-codika-staging.web.app Browser pour la voter page : on ouvre l'URL /vote/<secret> récupérée de l'inbox de test ou via owner vote-link, on clique « Pour », on vérifie que Firestore reflète le vote via CLI.
  • DATA copro-test-qa-2026-05-09 Copropriété scratch sur staging, créée en début de passe, archivée à la fin. Toutes les opérations CRUD vivent dedans. On ne touche jamais de données client réelles (IGS ou autre).
05 · La séquence de test (Brugmann replay)

Le scénario canonique, étape par étape.

On reprend le scénario Brugmann de cli.html §07 et on l'adapte pour le test : version réduite (5 lots / 5 owners / 3 résolutions au lieu de 166 / 162 / 5), 2 votants réels via magic link au lieu d'un script qui poke syndicCastVote. Tout le reste est identique. Le syndic, c'est un compte Firebase de test qu'on a configuré comme membre de la copro-scratch.

# ━━━━━━━━ SETUP — login syndic + scratch copro + 5 lots + 5 owners ━━━━━━━━ $ syndicable login:request --email qa-syndic@codika.io --profile staging $ syndicable login:verify --email qa-syndic@codika.io --code 847291 ✓ Profile "staging" persisted $ CID=$(syndicable copro create --legal-name "Copro QA Test" --bce 0700.000.001 \ --address-line1 "rue de Test 42" --postal-code 1000 --city Bruxelles \ --country BE --json | jq -r .coproId) $ for i in 1 2 3 4 5; do syndicable lot create --copro-id $CID --reference "L00$i" \ --quotities "GEN=200" --lot-type apartment done # Total GEN = 1000 (égal au expectedTotal de la clé GEN par défaut). $ for i in 1 2 3 4 5; do syndicable owner create --copro-id $CID \ --display-name "Owner $i" \ --email "syndicable-test+owner$i@example.com" done $ syndicable ownership assign --copro-id $CID ... # 5 assigns L00i ↔ owner i # ━━━━━━━━ AG DRAFT — assembly create + 3 résolutions + show ━━━━━━━━ $ AID=$(syndicable assembly create --copro-id $CID --title "AGO QA Test 2026" \ --scheduled-at 2026-05-15T14:00 --mode hybrid --default-language fr --json | jq -r .assemblyId) $ syndicable assembly resolution add --copro-id $CID --assembly-id $AID \ --title "R1 — Approbation comptes" --threshold absolute --key-id GEN $ syndicable assembly resolution add --copro-id $CID --assembly-id $AID \ --title "R2 — Décharge syndic" --threshold absolute --key-id GEN $ syndicable assembly resolution add --copro-id $CID --assembly-id $AID \ --title "R3 — Travaux toiture" --threshold four_fifths --key-id GEN $ syndicable assembly show --copro-id $CID --assembly-id $AID status: draft · 3 resolutions · 5 owners (5 lots covered) · 0 tokens # ━━━━━━━━ CONVOCATION — send + verify deliveries ━━━━━━━━ $ syndicable convocation send --copro-id $CID --assembly-id $AID --override-15-day-warning ✓ Convocation sent · 5 tokens issued · 5 emails queued $ syndicable convocation list --copro-id $CID --assembly-id $AID 5 tokens · 5 deliveries (queued: 0 · sent: 5 · bounced: 0) # Owner 1 et Owner 2 ouvrent leur magic link dans un browser réel. # attendance.join déclenchée automatiquement par la voter page (state: present_online). $ syndicable attendance show --copro-id $CID --assembly-id $AID --watch [poll t+0s] present_online: 0 in_person: 0 not_joined: 5 [poll t+4s] present_online: 2 in_person: 0 not_joined: 3 ^C $ syndicable assembly resolution open --copro-id $CID --assembly-id $AID --resolution-id R001 # Owners 1 et 2 cliquent « Pour » dans la voter page → castVote. # Owners 3, 4, 5 absents → resteront not_voted, abstention auto à la clôture. $ syndicable vote show --copro-id $CID --assembly-id $AID --resolution-id R001 --watch [poll t+2s] for: 400 against: 0 abstain: 0 not_voted: 600 $ syndicable assembly resolution close --copro-id $CID --assembly-id $AID \ --resolution-id R001 --confirm-auto-default ✓ R001 closed · for=400 against=0 abstain=600 → REJECTED (absolute, <500) # Idem R002 et R003 (R003 = four_fifths, échec attendu avec 400/1000). $ syndicable assembly end --copro-id $CID --assembly-id $AID ✓ Assembly ended # ━━━━━━━━ PV — generate + edit + export PDF + distribute ━━━━━━━━ $ syndicable pv generate --copro-id $CID --assembly-id $AID $ syndicable pv update-text --copro-id $CID --assembly-id $AID \ --section presidentSecretaryNotes --value "Test QA — pas de remarque." $ syndicable pv export --copro-id $CID --assembly-id $AID --output /tmp/pv-qa.pdf # Inspection manuelle : ouvrir /tmp/pv-qa.pdf, vérifier présence/exceptions/seuils. $ syndicable pv distribute --copro-id $CID --assembly-id $AID ✓ PV distributed to 5 owners # Vérif : 5 e-mails de PV dans Resend, PDF en pièce jointe ou via lien signé. # La passe complète tient en ~3 minutes de wall-clock si tout va bien. Si quelque chose casse, on saute en §07 (boucle fix-and-retry). La portion browser (les deux owners qui votent via magic link) est manuelle ; l'agent QA ouvre deux fenêtres incognito, copie/colle l'URL /vote/<secret> reçue dans Resend, clique. Une fois ce comportement validé, on peut basculer sur syndicCastVote pour accélérer les itérations suivantes.

06 · La table des cas négatifs

Vingt scénarios « try-to-break-it ».

Pour chaque famille de CF, on essaie de violer une garde — auth, état, validation métier. Le résultat attendu est un HttpsError typé propre (jamais un internal, jamais une stack trace brute). Le code de sortie CLI est mappé depuis le code d'erreur (unauthenticated → 1, permission-denied → 2, failed-precondition → 3, not-found → 4, invalid-argument → 5).

# Cloud Function Scénario négatif Code attendu Exit Critère pass/fail
01 castVote Token révoqué via revokeAccessToken juste avant le cast. permission-denied 2 Pass si l'erreur est typée et le vote n'apparaît pas dans Firestore.
02 castVote Token expiré (Assembly status = ended). failed-precondition 3 Pass si message d'erreur explique « assembly is no longer live ».
03 castVote Vote sur résolution avec quotitiesConcerned=0 pour le lot. failed-precondition 3 Pass si erreur claire « no concerned quotity for this lot ».
04 castVote Choix invalide (ex. --choice maybe au lieu de for/against/abstain). invalid-argument 5 Pass si refus avant tout side-effect Firestore.
05 closeResolution Clôturer une résolution dont le statut est pending (jamais ouverte). failed-precondition 3 Pass si refus + état Firestore inchangé.
06 closeResolution Clôturer sans --confirm-auto-default alors que not_voted > 0. failed-precondition 3 Pass si erreur indique le besoin de confirmer l'abstention auto.
07 openResolution Ouvrir une résolution sur une assembly draft (jamais démarrée). failed-precondition 3 Pass si message « assembly not live ».
08 sendConvocation Envoyer une convocation < 15 jours avant la date sans --override-15-day-warning. failed-precondition 3 Pass si refus + indication explicite du flag à passer.
09 sendConvocation Envoyer la convocation 2 fois pour la même AG. failed-precondition 3 Pass si refus + indication « already convoked ».
10 getMyVoteLink Appel pour un Owner archivé (archived: true). not-found 4 Pass si refus + Firestore non muté (pas de nouveau token créé).
11 getMyVoteLink Appel par un user qui n'est pas membre de la copro. permission-denied 2 Pass si refus + log de tentative d'accès.
12 validateToken Token avec un seul caractère altéré (typo). not-found 4 Pass si refus en temps constant (pas de timing attack possible).
13 joinAssembly Rejoindre une assembly convoked (pas encore live). failed-precondition 3 Pass si refus avec message « assembly not started yet ».
14 heartbeat Heartbeat sans joinAssembly préalable (pas d'Attendance doc). not-found 4 Pass si refus + pas de side-effect Firestore.

19 archiveOwner Archiver un Owner qui détient encore un Lot non transféré. failed-precondition 3 Pass si refus + indication des lots toujours rattachés.

21 issueProcurationToken Émettre une procuration pour un Owner déjà present_in_person. failed-precondition 3 Pass si refus + message « owner already attending in person ».

22 lignes — un peu plus que les 20 cibles, et c'est bien : la marge couvre les combinaisons rares (procuration + in-person, par exemple). Si un cas surface une erreur internal ou un 500, c'est un bug : on note la stack, on file en §07.

07 · Le fix-and-retry loop

Six étapes, jusqu'au vert.

La discipline du loop, c'est de garder chaque tour court (~5-15 min). Si un fix demande plus, on sort du loop, on ouvre une issue, et on continue le reste de la passe sans cette CF. Le but est de terminer la passe, pas de tout débugger en chaîne.

# 1. Test fails — capter le contexte avant de toucher quoi que ce soit. $ syndicable assembly resolution close --copro-id $CID --assembly-id $AID \ --resolution-id R001 --confirm-auto-default ✗ FAIL · HttpsError code=internal · Cannot read property 'tally' of undefined # 2. Diagnose — logs CF + état Firestore. $ gcloud functions logs read closeResolution \ --project=syndicable-codika-staging --gen2 --region=europe-west1 --limit 20 $ syndicable assembly resolution show --copro-id $CID --assembly-id $AID --resolution-id R001 --json # Diagnostic : tally absent du doc (pas de votes castés → aggregator skippé). # 3. Fix — patch dans syndicable-app, build, redéploy ciblé. $ cd ~/.codika/agency/projects/syndicable/syndicable-app/functions $ # éditer src/features/assemblies/services/resolutionService.ts → init tally à 0,0,0 $ npm run build $ firebase deploy --only functions:closeResolution --project syndicable-codika-staging ✓ Function closeResolution deployed (4.3 s, region europe-west1) # 4. Re-run — relancer juste le scénario qui a cassé. $ syndicable assembly resolution close --copro-id $CID --assembly-id $AID \ --resolution-id R001 --confirm-auto-default ✓ R001 closed · for=400 against=0 abstain=600 → REJECTED # 5. Itérer — passer au scénario suivant. Bundling des fixes en fin de passe. $ git add -A && git commit -m "fix(qa): init tally on closeResolution when no votes cast" # 6. Final — bundle des fixes en une PR à la fin. $ gh pr create --title "fix(qa): post-deploy testing sweep" --body "..."

La règle de bundling : une seule PR fix(qa): post-deploy testing sweep à la fin si l'ensemble des fixes est cohérent (≤10 fichiers, ≤300 LOC). Au-delà, on fragmente : un PR par feature touchée. Le journaling se fait dans docs/decisions-log.md si un fix change un contrat de CF.

08 · Connue à fixer en priorité

Cinq échecs Phase D dans assemblies.test.ts — à fixer avant tout le reste.

Sur main, npm test -- assemblies.test.ts échoue sur 5 cas Phase D liés au scénario Brugmann. La cause racine est une collision arithmétique sur la clé GEN par défaut : le code de Phase D a hérité d'un expectedTotal=1000 pour la clé GEN, mais les fixtures Brugmann répartissent des quotités >200 par lot sur 5 lots → total > 1000 → la validation refuse.

Symptôme

Tous les tests Phase D Brugmann (5 sur 5) échouent avec une assertion du genre Expected total quotity ≤ 1000, got 1xxx. Le test runner remonte FAIL src/features/assemblies/__tests__/assemblies.test.ts avec les 5 assertions consécutives.

Fix attendu

Deux options, à trancher par le sub-agent quand il aura lu le code : (a) les fixtures du test utilisent expectedTotal: 10000 pour la clé GEN au lieu de 1000, ce qui aligne sur des quotités plausibles (200/lot × 5 lots = 1000, mais Brugmann réel est calé sur 10000) ; (b) les résolutions du fixture utilisent des quotités plus basses pour rentrer sous 1000. Recommandation : option (a) — c'est plus représentatif du métier (les vraies copros tournent à 10000).

Vérification

cd functions && npm test -- assemblies.test.ts doit passer 100% (zero failure, zero skip non-explicite). Une fois le fix posé, c'est la première chose à commiter — toute la suite de la passe dépend de cette baseline propre.

09 · Phasage du test

Deux phases — la sentinelle, puis le négatif.

On sépare proprement le happy path du negative testing pour deux raisons : (a) la sentinelle est rapide et tient lieu de baseline avant tout négatif ; (b) le negative dépend que la sentinelle passe (sinon on confond bug et environnement). Pas de raison de fragmenter plus.

TEST·F

Setup + happy path Brugmann

Setup (scratch copro sur staging, 5 lots, 5 owners, AG, 3 résolutions) + le scénario complet de §05 bout-en-bout. Objectif : le flux entier tourne sans erreur. Première chose : fixer les 5 échecs Phase D pré-existants de assemblies.test.ts — cf. §08. Sortie attendue : Brugmann replay vert, PV PDF dans /tmp, 5 e-mails dans Resend.

TEST·G

Negative testing sweep

22 scénarios du tableau §06 exécutés un par un, chacun sur la même copro scratch. Pour chaque ligne : on déclenche la condition, on vérifie le code d'erreur, on note le résultat. Objectif final : 100% des scénarios produisent le code d'erreur attendu. Tout écart est un bug — diagnostic + fix dans la boucle §07.

TEST·F = ~30 min wall-clock + temps de fix des 5 tests Phase D. TEST·G = ~1-2 h selon le nombre de fixes nécessaires.

10 · Critères d'acceptation

Sept conditions à cocher pour clôturer la passe.

On est verts quand chacun de ces critères est satisfait. Pas de demi-passe. Si un critère résiste après deux itérations de fix, on remonte au reviewer plutôt que de boucler indéfiniment.

  • Brugmann replay end-to-end tourne sans erreur inattendue (CLI + browser pour les votes mode token).
  • Les 5 échecs Phase D pré-existants sont fixés ; npm test -- assemblies.test.ts passe 100%.
  • Les 22 scénarios négatifs §06 produisent chacun le code d'erreur attendu, ni plus ni moins.
  • Le PV PDF généré pour la copro scratch matche structurellement le sample IGS (numéros de présence, exceptions nommées, métadonnées de seuil — cf. article.html §06).
  • La rotation de magic link via getMyVoteLink invalide le token précédent en ≤ 2 s (vérifié en castant un vote avec l'ancien token : permission-denied).
  • Le cleanup d'attendance stale (cleanupStaleAttendance) flippe une attendance « onglet ouvert mais inactif » à left en ≤ 3 min après le dernier heartbeat.
  • Le journal d'activité contient au moins une entrée pour chaque scénario testé (création AG, send convocation, open R, cast vote, close R, end AG, generate PV, distribute PV) — format consistent.

12 · Colophon

Ce qu'est ce plan — et ce qu'il n'est pas.

C'est un plan d'exécution de QA pour V1.1 + V1.1.5 + CLI. Le plan est statique ; l'exécution est confiée à un sub-agent qui prend cet article comme spec, déroule §05 puis §06, et applique §07 chaque fois qu'un test casse. La sortie attendue de l'exécution : tous les critères §10 cochés, plus une PR fix(qa): post-deploy testing sweep si des fixes ont été produits.

Ce que ce plan ne traite pas : tout ce qui est dans la colonne OUT du §02 (load, pen-test, RGPD, multi-region, migration de données IGS legacy).
