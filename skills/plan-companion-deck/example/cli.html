Version V1.1+CLI Module · Surface CLI 2026-05-09

Toutes les fonctions Cloud, accessibles depuis le terminal.

V1.0 a livré 23 wrappers callables dans le CLI. V1.1 ajoute ~25 Cloud Functions (assemblées, convocations, session live, PV) ; V1.1.5 en ajoute une dernière (getMyVoteLink). Ce plan dit lesquelles passent dans le CLI, sous quels noms, avec quels flags, et sous quel régime d'authentification.

01 · La thèse

Si une fonction Cloud existe, on doit pouvoir l'appeler depuis le terminal.

Le CLI syndicable existe pour trois usages : scripter les tâches répétitives (création de copro, import de lots, smoke test bout-en-bout), tester les Cloud Functions sans devoir cliquer dans la PWA, et déboguer les opérations de prod / staging quand l'UI ne suffit pas. V1.0 a tenu cette promesse pour les 23 fonctions livrées en Phase 5–14. V1.1 multiplie la surface par deux. Si on ne suit pas, le CLI devient un musée.

Le test mental est simple : un agent LLM ou un opérateur humain doit pouvoir, en enchaînant des commandes syndicable, dérouler une AG complète sur staging — de la création de l'assemblée à la distribution du PV — sans jamais ouvrir un navigateur. C'est ce que fait test:happy-path aujourd'hui pour V1.0 ; ce plan définit son équivalent pour V1.1.

Le persona qui ancre ce travail, c'est l'agent de support qui doit reproduire un bug de Brugmann à 22h. Si chaque CF a son wrapper CLI, l'agent n'a besoin que d'un terminal et de son token de login. Si la moitié manque, il doit ouvrir la PWA, se connecter, naviguer, cliquer — pendant que le client attend. L'investissement d'aujourd'hui, c'est l'autonomie opérationnelle de demain.

02 · Périmètre

Ce qui sort en V1.1+CLI — et ce qui ne sort pas.

Pas de surprise ici : on suit la liste exhaustive des nouveaux exports de functions/src/index.ts (V1.1 phases A à E + V1.1.5 phase G) et on ajoute quelques commandes de lecture quand elles rendent service. Tout le reste est différé.

Inclus · livré en V1.1+CLI

  • Un wrapper par CF nouvelle de V1.1 (~25) + V1.1.5 (1)
  • Quelques commandes de lecture utiles : assembly show, assembly list, vote show, attendance show
  • Deux modes d'authentification : login persistant (syndic) + flag --token (public)
  • Trois idiomes de sortie : humain par défaut, --json pour scripts, --watch pour les flux temps réel
  • Profile multi-staging cohérent avec V1.0 (--profile sur toutes les commandes)
  • Mise à jour du registre CF dans src/lib/functions.ts (+ types DTO inlined dans src/types/syndicable-app-types.ts)
  • Smoke test test:assembly-happy-path — équivalent V1.1 du test:happy-path existant

Hors · pas dans V1.1+CLI

  • Flux scriptés multi-CF (« lance une AG complète en une commande ») — l'utilisateur les compose en bash
  • Wrappers pour les triggers Firestore (onAssemblyWritten, onResolutionWritten, onVoteWritten, cleanupStaleAttendance) — non callables
  • heartbeat en commande dédiée — usage interne du voter PWA, pas un cas-d'usage CLI
  • Service-account auth — Firebase user OTP uniquement, comme V1.0
  • REPL interactif, scripts de complétion shell — déjà OUT en V1.0
  • Tests unitaires par commande — on s'appuie sur le smoke happy-path
  • Réception d'e-mails de convocation côté CLI — on assume que l'opérateur va lire son inbox
03 · Architecture

Trois choix dont le reste du plan découle.

Trois bifurcations actées, qui coupent court aux discussions de design pendant l'implémentation. Le phasage et l'annexe partent de l'hypothèse que ces trois choix sont pris.

CHOIX 01

Mirroir 1:1 du backend.

Une commande CLI par Cloud Function. Pas d'orchestration côté CLI ; pas de macros qui appellent trois CFs en chaîne. Si l'utilisateur veut « envoyer la convocation et ouvrir la première résolution dans la même action », il écrit deux lignes de bash — le CLI ne se substitue pas à un agent.

Avantage : zéro surface d'intelligence côté CLI. Quand le backend renomme un paramètre, on met à jour le wrapper et c'est tout. Pas de macro à débugger, pas de divergence sémantique. C'est exactement le pattern de V1.0 — on le prolonge.

CHOIX 02

Deux modes d'authentification cohabitent.

Le CLI gère deux régimes d'auth qui ne se mélangent jamais. Mode A (syndic) : login OTP persisté dans ~/.config/syndicable/config.json, ID token attaché automatiquement à chaque appel. C'est l'existant V1.0, intact. Concerne tout ce qui est syndic-only : createAssembly, sendConvocation, startAssembly, generatePvDraft...

Mode B (token public) : flag explicite --token <secret> sur la ligne de commande. Pas de login requis, pas de persistance. Concerne les CFs de la voter page : validateToken, joinAssembly, leaveAssembly, castVote. Le secret transite en clair dans l'historique shell — ce n'est pas un secret long-terme, c'est le même genre de jetable que la magic link e-mail.

Question ouverte (§09 q01) : doit-on permettre au mode A d'invoquer aussi castVote au nom du syndic, sans --token ? Il existe déjà syndicCastVote côté backend pour ça — donc la réponse de bon sens est non, on garde la frontière nette.

CHOIX 03

Sortie : humain / JSON / watch.

Trois idiomes de sortie, appliqués partout où c'est pertinent. --json existe déjà sur toutes les commandes V1.0 ; on le garde. Nouveau pour V1.1 : --watch sur les commandes de lecture qui suivent un état évolutif — assembly show --watch, vote show --watch, attendance show --watch. Sous le capot, polling Firestore REST toutes les 2 secondes (pas de listener SDK long-lived — on reste cohérent avec callable.ts qui n'utilise que fetch).

Le polling REST n'est pas du « vrai » streaming, mais c'est suffisant pour observer une session AG en temps semi-réel depuis le terminal. La latence de 2 s est plus faible que la fréquence à laquelle un syndic ferait F5 dans le navigateur de toute façon. Question ouverte (§09 q02) : faudra-t-il passer à un listener SDK plus tard ? Probablement, si on observe des décalages en charge.

05 · La table des commandes

Mapping CF → CLI, exhaustif.

Une ligne par Cloud Function nouvelle ou commande de lecture. Les flags listés sont les principaux ; --profile et --json sont implicites partout. Auth : syndic = login persisté requis ; token = flag --token requis ; either = les deux marchent (ex. syndic-cast) ; read = lecture Firestore REST (login persisté).

06 · Auth + flags — exemples concrets

Six exemples qui couvrent les six idiomes du CLI.

Login OTP, création d'AG, envoi de convocation avec override 15 jours, vote token, génération + export du PV, lecture watch. Tout ce que l'agent de support nocturne aura besoin de connaître.

# 1) Login OTP — V1.0, inchangé. $ syndicable login --email kd@igsyndic.be Code OTP envoyé à kd@igsyndic.be. Saisir le code : 123456 ✓ Profile "default" (kd@igsyndic.be), uid: T9k... # 2) Création d'AG (syndic-only, login persisté). $ syndicable assembly create \ --copro-id brugmann-fr \ --title "AGO 2026 — Brugmann" \ --scheduled-at 2026-06-12T17:30 \ --mode hybrid \ --default-language fr ✓ Assembly created: a_01HX9... # 3) Envoi de la convocation, avec override de l'avertissement 15 jours. $ syndicable convocation send \ --copro-id brugmann-fr \ --assembly-id a_01HX9... \ --override-15-day-warning ✓ Convocation sent · 166 tokens issued · 166 emails queued # 4) Vote en mode token public (le secret arrive par e-mail). $ syndicable vote cast \ --token 5b8...d2f \ --assembly-id a_01HX9... \ --resolution-id R001 \ --lot-id A12-B \ --choice for ✓ Vote cast · A12-B · R001 · for # 5) PV : génération, édition, export local. $ syndicable pv generate --copro-id brugmann-fr --assembly-id a_01HX9... ✓ PV draft created · 18 resolutions imported $ syndicable pv export \ --copro-id brugmann-fr \ --assembly-id a_01HX9... \ --output /tmp/pv-brugmann-2026.pdf ✓ PDF exported to /tmp/pv-brugmann-2026.pdf (4.2 MB) # 6) Lecture en mode watch — utile pendant la session live. $ syndicable assembly show --copro-id brugmann-fr --assembly-id a_01HX9... --json --watch { "status": "live", "currentResolutionId": "R003", "quorumIn": 0.62, "quorumOut": 0.71, "attendance": { "online": 23, "in_person": 41, "left": 5 } } [poll t+2s] (no change) [poll t+4s] currentResolutionId: R003 → R004 ...

Patterns récurrents. Toute commande syndic prend --copro-id (ou utilise le défaut configuré). Toute commande sur une AG prend --assembly-id ; toute commande sur une résolution ajoute --resolution-id. Les flags qui éditent du free-text long acceptent --from-file en alternative à un --content mono-ligne. Les flags d'override critiques (15-day-warning, force-close, etc.) sont nommés explicitement et n'ont pas de valeur par défaut implicite.

Sortie. Par défaut, ligne d'OK colorée + résultat lisible. Avec --json, JSON strict sur stdout (parseable, scriptable). Avec --watch, polling 2 s ; signal SIGINT (Ctrl-C) pour arrêter ; chaque tick imprime soit un diff, soit un (no change) en gris.

07 · Brugmann depuis le CLI

Une AG complète, du terminal, sans jamais ouvrir le navigateur.

Pour vérifier que la surface CLI tient debout, on déroule l'AG Brugmann 2026 entièrement depuis le shell. Login, création, agenda, convocation, démarrage, vote, fermeture, PV, distribution. Une seule session bash, une seule AG.

# Setup — login en tant que syndic. $ syndicable login:request --email kd@igsyndic.be $ syndicable login:verify --email kd@igsyndic.be --code 847291 ✓ Profile "default" persisted # 1. Créer l'AG, ajouter 5 résolutions, réordonner. $ AID=$(syndicable assembly create --copro-id brugmann --title "AGO 2026" \ --scheduled-at 2026-06-12T17:30 --mode hybrid --default-language fr --json | jq -r .assemblyId) $ syndicable assembly resolution add --copro-id brugmann --assembly-id $AID \ --title "Approbation comptes 2025" --threshold absolute --key-id GEN $ syndicable assembly resolution add --copro-id brugmann --assembly-id $AID \ --title "Décharge syndic" --threshold absolute --key-id GEN $ syndicable assembly resolution add --copro-id brugmann --assembly-id $AID \ --title "Budget 2026" --threshold absolute --key-id GEN $ syndicable assembly resolution add --copro-id brugmann --assembly-id $AID \ --title "Réfection toiture (~80k€)" --threshold four_fifths --key-id GEN $ syndicable assembly resolution add --copro-id brugmann --assembly-id $AID \ --title "Rapport CC (info)" --threshold information_only # 2. Envoyer la convocation à 166 propriétaires. $ syndicable convocation send --copro-id brugmann --assembly-id $AID ✓ Convocation sent · 166 tokens issued · 166 emails queued # 3. Démarrer l'AG le jour J. $ syndicable assembly start --copro-id brugmann --assembly-id $AID ✓ Assembly is now LIVE # 4. Ouvrir R1, observer les votes en temps réel, clôturer. $ syndicable assembly resolution open --copro-id brugmann --assembly-id $AID --resolution-id R001 $ syndicable vote show --copro-id brugmann --assembly-id $AID --resolution-id R001 --watch [poll t+0s] for: 0 against: 0 abstain: 0 not_voted: 166 [poll t+2s] for: 1240 against: 0 abstain: 0 not_voted: 165 [poll t+12s] for: 5720 against: 4050 abstain: 230 not_voted: 0 ^C $ syndicable assembly resolution close --copro-id brugmann --assembly-id $AID --resolution-id R001 ✓ R001 closed · for=5720 against=4050 abstain=230 → PASSED # 5–8. Idem pour R002..R005, puis terminer l'AG. $ syndicable assembly end --copro-id brugmann --assembly-id $AID ✓ Assembly ended at 2026-06-12T20:14Z # 9. Générer + distribuer le PV. $ syndicable pv generate --copro-id brugmann --assembly-id $AID $ syndicable pv export --copro-id brugmann --assembly-id $AID --output /tmp/pv-brugmann-2026.pdf $ syndicable pv distribute --copro-id brugmann --assembly-id $AID ✓ PV distributed to 166 owners

Le scénario tient en une trentaine de lignes de bash. Si test:assembly-happy-path (smoke test scripté) reproduit ces étapes en mode JSON sans intervention humaine, la surface CLI est validée. C'est le seul critère go/no-go de la phase CLI·F.

Ce que ce scénario ne montre pas : le mode token (utilisé par les copropriétaires eux-mêmes — pas par le syndic) et la commande owner vote-link de V1.1.5 (qui s'invoque depuis un compte copropriétaire authentifié, pas depuis le compte syndic). Les deux entrent en jeu uniquement quand on simule un copropriétaire end-to-end avec son propre profil --profile.

08 · Phasage

Deux phases d'implémentation, alignées sur le backend.

Le CLI suit le backend de près mais pas trop. On groupe les wrappers en deux vagues : la grosse (CLI·F, en parallèle de l'arrivée de Phase D côté backend) et la petite (CLI·G, après la Phase E + la V1.1.5). Pas de raison de fragmenter plus.

CLI·F

Wrappers A → C + lectures D

19 wrappers (createAssembly..syndicCastVote, hors PV) + 5 commandes de lecture (assembly show/list, resolution show, attendance show, vote show) + le smoke test. Vérification : test:assembly-happy-path --cleanup exécute le scénario Brugmann en JSON sans erreur, en moins d'une minute, et nettoie ses traces. Mise à jour des types DTO + du registre CF.

CLI·G

Wrappers E + V1.1.5

4 wrappers PV (generate, update-text, export, distribute) + 1 wrapper V1.1.5 (owner vote-link). Plus petit, plus tard. Vérification : depuis le scénario Brugmann CLI·F, on peut maintenant exécuter pv generate → pv export → pv distribute en bout de chaîne, et dans une session séparée logguée comme copropriétaire, owner vote-link retourne une URL /vote/<secret> valide.

CLI·F = ~3-4 jours d'un implémenteur concentré (24 commandes). CLI·G = ~1 jour (5 commandes).

09 · Questions ouvertes

Sept points à trancher avant ou pendant l'implémentation.

Aucune ne bloque CLI·F. Chacune se résout au moment où la commande qui la rencontre est codée. La recommandation est ma meilleure lecture actuelle ; la décision finale revient au reviewer.

# Question Recommandation
01 Le mode A (login persisté) doit-il aussi accepter vote cast sans --token, pour faciliter le test syndic ? Non. syndicCastVote existe précisément pour ça côté backend. On garde la frontière nette : vote cast = mode token public exclusivement.
02 Le flag --watch utilise-t-il un listener Firestore SDK ou du polling REST ? Polling REST 2 s par défaut. Reste cohérent avec l'architecture callable.ts (fetch-only). Un listener SDK long-lived demanderait de garder le module Firebase Web SDK chaud — pas un investissement justifié pour V1.1.
03 pv export : sauvegarder en local (--output) ou retourner l'URL signée ? Les deux. Par défaut, URL signée sur stdout (cohérent avec doc url). Avec --output <path>, on télécharge via fetch + écrit le PDF localement. Plus pratique pour les agents.
04 Faut-il une commande convocation list qui affiche tous les AccessTokens d'une AG (pour debug bounces) ? Oui, à ajouter dans CLI·F. C'est la seule façon raisonnable de répondre à « pourquoi tel owner n'a pas reçu sa convocation ». Lecture Firestore REST + masquage des secrets.
05 Doit-on stocker le --copro-id par défaut dans config.json pour ne plus le retaper ? Oui, via une commande syndicable use-copro <id> (ou un flag de use). Évite de polluer chaque ligne. À ajouter en CLI·F si le temps le permet, sinon en CLI·G.
06 Le smoke test test:assembly-happy-path doit-il aussi tester le mode token (vote cast avec un secret extrait du e-mail) ? Non en CLI·F, oui en CLI·G. Récupérer un secret de magic link demande soit un email-fixture soit une lecture directe d'AccessToken (admin SDK). On utilise syndicCastVote dans CLI·F et on ajoute le test mode-token quand le test framework du backend le permet.
07 Le wrapper convocation validate est-il vraiment utile, ou pure dette ? Utile pour debug : « ce token marche-t-il toujours ? est-ce qu'il a été révoqué ? ». À garder dans CLI·F mais marqué « debug-only » dans la description. Pas dans le smoke test.

Ce qu'est ce plan — et ce qu'il n'est pas.

C'est la conception de la surface CLI pour V1.1 + V1.1.5. Ce n'est pas un patch — aucun fichier de syndicable-cli/ n'est modifié par sa lecture. Ce n'est pas un engagement de livraison à N jours — le phasage suppose un implémenteur concentré, et chaque phase peut respirer si le backend bouge.

Ce que cet article fige : la forme. Six nouveaux groupes de commandes, 28 lignes dans la table CF→CLI, deux modes d'authentification, trois idiomes de sortie. Le code commence à la validation. Si quelque chose dans la forme cloche — un nom de groupe, une frontière d'auth, un flag manquant, une question ouverte mal tranchée — c'est maintenant qu'il faut le dire, avant que CLI·F ne démarre.

Ce qui n'est pas couvert : tout ce qui est dans la colonne OUT du §02. Les flux scriptés multi-CF, les wrappers de triggers, le service-account auth, le REPL. Autant de pistes pour V1.2 ou plus tard. CLI·F + CLI·G livrent le minimum suffisant pour qu'un agent de support nocturne puisse dérouler une AG complète sans ouvrir le navigateur — et c'est exactement la barre qu'on cherche.
