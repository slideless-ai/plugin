Version V1.1.5 Module · Vue copropriétaire 2026-05-09

L'AG, vue depuis le canapé du copropriétaire.

Les phases A à C ont livré le côté syndic ; V1.1.5 livre le côté copropriétaire. Une seule vue cross-immeubles, le détail AG en lecture seule, et un bouton « Rejoindre » qui ouvre la magic link déjà reçue par e-mail.

01 · La thèse

Le copropriétaire devient un utilisateur de plein droit du dashboard.

Les phases A, B et C de V1.1 ont livré le syndic de bout en bout : rédaction d'AG, envoi des convocations avec magic link, animation de la session en direct, brouillon de PV. Le flux légal est en place — côté syndic. Aujourd'hui, un copropriétaire qui se connecte à Syndicable voit le même NavAdministration, la même liste d'AG côté syndic, le même détail AG bardé de boutons d'admin, sans cadrage « qu'est-ce que cette AG veut dire pour moi en tant que votant ».

V1.1.5 fait basculer le copropriétaire de destinataire d'e-mail à utilisateur de premier ordre du dashboard, sans construire un portail parallèle. Une seule app, deux visages. Le syndic continue d'orchestrer ; le copropriétaire reçoit, consulte, et rejoint depuis la même interface.

Le persona qui ancre ce travail, c'est l'IGS / Brugmann Ferme Rose : 166 copropriétaires, beaucoup avec des lots dans plusieurs immeubles à Bruxelles. Aujourd'hui, ce copropriétaire-là doit basculer l'OrganizationSwitcher trois fois pour savoir quelle AG arrive en premier. Demain, il ouvre le popover du sidebar et voit ses trois AG dans une seule liste, triées par état.

V1.1.5 est purement additif. Pas de migration de données, pas de changement de schéma Firestore, aucun risque sur le côté syndic. Un store, une section de popover, une bascule viewMode dans une page existante, et une callable. Cinq petites phases. Le code commence à la validation de cet article.

02 · Périmètre, figé

Ce qui sort en V1.1.5 — et ce qui ne sort pas.

La règle du jeu de V1.1 reste : tout ce qui n'est pas dans la colonne IN est mis en pause. La colonne OUT n'est pas un rejet, c'est une file d'attente — V1.2 ou plus tard.

Inclus · livré en V1.1.5

  • Nouvelle section « Mes assemblées » dans le popover de l'OrganizationSwitcher (cross-org)
  • Mode lecture seule de /organizations/[orgId]/assemblies/[aId] pour userRole === 'member' ou 'viewer'
  • Table des résolutions vue copropriétaire pour les AG clôturées (totaux en quotités, sans nominatifs)
  • CTA « Rejoindre la session » qui ouvre la magic link depuis le dashboard
  • Le copropriétaire voit le statut de vote de ses propres lots, par résolution
  • FR + EN, parité dark/light, identique à V1.1
  • Tri du popover : live en premier, puis convoked ascendant, puis ended descendant
  • Limite à 5 AG visibles + lien « voir tout » pour les copropriétaires en multi-immeuble

Hors · différé en V1.2 ou plus tard

  • Route dédiée /my-assemblies — la liste vit dans le popover
  • Notifications push « votre AG démarre dans 1h »
  • Téléchargement du PV in-app (déjà couvert par le Document historic_pv)
  • Vues archive d'historique d'AG (« mes 5 dernières AG »)
  • Rédaction de résolution côté copropriétaire
  • Soumission de points d'ordre du jour §3 par les copropriétaires (déjà OUT en V1.1)
  • Surface dashboard pour le titulaire d'une procuration sans compte Syndicable
  • Affichage des AG annulées dans le popover
03 · Trois choix d'architecture

Les décisions dont le reste de V1.1.5 découle.

Trois bifurcations. Trois décisions désormais arrêtées. Le phasage et l'annexe partent de l'hypothèse que ces trois choix sont actés.

CHOIX 01

Une page, deux visages.

On réutilise /organizations/[orgId]/assemblies/[aId] pour le copropriétaire. Pas de route parallèle, pas de squelette dupliqué. Chaque bouton de mutation est gaté par hasMinimumRole(userRole, 'admin'). La page dérive un viewMode du rôle utilisateur : 'syndic' | 'owner'.

La skin owner-side est principalement une soustraction, pas un ajout — on retire les boutons d'admin et on remplace deux blocs (la table des résolutions, le panneau « Mon vote »). Une route, deux audiences. La maintenance reste centralisée.

CHOIX 02

Store cross-org calqué sur myRepresentedLotsStore.

Un nouveau myAssembliesStore lit (a) les organisations de l'utilisateur via userOrganizationsStore, (b) les Owner docs représentés via collectionGroup('owners') avec representedBy array-contains uid, et fan-out une petite query assemblies par copro : status in ['convoked', 'live', 'ended'], orderBy('scheduledAt', 'desc'), limit(5).

Forme à deux étages avec un seul unsub retourné — la même que myRepresentedLotsStore, déjà éprouvée en Phase 14. Pas de nouvelle invention d'architecture, juste un instance de plus du pattern « cross-org derived ».

CHOIX 03

Le bouton « Rejoindre » fait tourner le token, pas une nouvelle route.

Le dashboard surface l'AccessToken déjà émis pour ce couple owner+assembly. Pour faire ça sans stocker le secret en Firestore, une nouvelle callable getMyVoteLink({ assemblyId }) régénère un secret frais côté serveur, fait tourner l'AccessToken de cet owner (révoque l'ancien), et retourne l'URL. Le frontend fait un window.open(url, '_blank').

Trade-off : le lien magique de l'e-mail continue de marcher jusqu'au premier clic dashboard ; après, il est révoqué. Ça correspond au substrat légal — un seul token actif par owner+AG — et ça donne au copropriétaire un chemin de réinitialisation propre s'il a perdu l'e-mail. L'alternative envisagée puis parquée : une route /organizations/[orgId]/assemblies/[aId]/vote authentifiée qui contourne les tokens — ajoute un second chemin d'auth qui doit rester en phase avec celui de la route publique. Pas la peine pour V1.1.5.

04 · Le popover, redessiné

Trois sections, une logique de rôle.

L'OrganizationSwitcher regroupe déjà aujourd'hui deux rôles : copropriétés dont l'utilisateur est syndic (admin) et lots dont il est propriétaire (member). V1.1.5 ajoute une troisième section — Mes assemblées — qui fusionne les AG des deux rôles dans une seule liste triée.

┌──────────────────────────────┐ │ 🏢 Copropriétés │ ← existant │ • Brugmann Ferme Rose │ │ • Résidence Albert │ │ │ │ ───────────────────────── │ │ 🏠 Mes lots │ ← existant (Phase 14) │ • Lot A12 — Brugmann │ │ │ │ ───────────────────────── │ │ 📅 Mes assemblées │ ← NOUVEAU │ ● Brugmann · 12 juin │ │ En cours · Copro. │ │ • Albert · 24 juin │ │ À venir · Syndic │ │ • Brugmann · 28 mars 2024 │ │ Terminée · Copro. │ │ │ │ ───────────────────────── │ │ ➕ Nouvelle copropriété │ ← existant └──────────────────────────────┘

Mockup de la troisième section. Le pictogramme ● en tête d'AG signifie « en cours, rejoignable à l'instant » ; le • pour les états convoked et ended.

Chaque ligne d'AG affiche quatre choses : la date abrégée, le nom de la copropriété, un badge de statut (à venir | en cours | terminée), et un pill de rôle (Syndic | Copropriétaire). Le tri suit l'urgence : AG live en premier (si l'utilisateur peut rejoindre maintenant), puis AG convoked par scheduledAt ascendant (la prochaine d'abord), puis AG ended par scheduledAt descendant (la dernière en haut).

La section affiche au plus 5 entrées, suivies d'un lien « voir tout » uniquement si l'utilisateur a plus de 5 AG actives ou récentes. Pour la majorité des copropriétaires (1 ou 2 immeubles, ~1 AG par an), la section reste compacte. Pour le syndic-multi-copros, elle reste navigable.

05 · Le store cross-org

myAssembliesStore.ts — le pattern à deux étages.

Fichier neuf à src/lib/services/assemblies/stores/myAssembliesStore.ts. Pattern derived au-dessus de userDataStore + userOrganizationsStore, exactement comme myRepresentedLotsStore. Deux étages, un seul unsub retourné.

Étage 1 : pour chaque organisation où l'utilisateur a un rôle (member, admin, syndic), on déclenche une query assemblies filtrée sur status in ['convoked', 'live', 'ended']. Étage 2 : pour les copros où l'utilisateur n'est pas membre mais est représentant d'un Owner (cas IGS-Brugmann typique), on lit la liste des Owner via collectionGroup('owners') + representedBy array-contains uid et on fan-out la même query d'assemblies. Les deux flux se fusionnent dans le store de sortie.

interface MyAssembly { id: string; coproprieteId: string; coproprieteName: string; scheduledAt: Timestamp; status: AssemblyStatus; // 'convoked' | 'live' | 'ended' myRoleInCopro: 'syndic' | 'owner'; // For 'live' AGs only — emitted lazily, not in the snapshot // (the magic-link rotation is a button-driven callable). }

Tri. Le store sort une liste plate, triée par (statusRank, scheduledAt) où statusRank est { live: 0, convoked: 1, ended: 2 }. Pour convoked on trie par date ascendante (la prochaine d'abord) ; pour ended, descendante (la dernière d'abord). Ça donne au popover l'ordre exact qu'il rend.

Dedup. Cas-limite : un syndic qui a aussi un lot dans la même copro remonterait deux fois (une via membership admin, une via Owner représenté). On dédup par (coproprieteId, assemblyId) et on garde le rôle le plus privilégié — 'syndic' l'emporte sur 'owner' dans la même copro. Une seule ligne, badge Syndic. C'est aussi la recommandation de la question ouverte 03.

06 · La page détail, mode lecture seule

Ce qui change dans [aId]/+page.svelte.

Pas de fork, pas de duplication. Le fichier src/routes/(dashboard)/organizations/[organizationId]/assemblies/[aId]/+page.svelte dérive un viewMode du userRole courant, et chaque action d'admin est gatée par un seul {#if viewMode === 'syndic'}. La structure de la page reste la même ; le squelette d'admin disparaît pour le copropriétaire.

Concrètement, voici ce qui devient invisible quand viewMode === 'owner' :

  • Boutons Pencil (éditer l'AG), XCircle (annuler), Plus (ajouter une résolution), Mail (envoyer la convocation), UserPlus (émettre une procuration).
  • Boutons « Démarrer l'AG », « Terminer l'AG ».
  • Contrôles ouvrir / clôturer / passer une résolution dans LiveAgendaTimeline.
  • Table des livraisons de convocation (infrastructure syndic).

Et ce qui change de comportement :

  • TableAgenda reçoit canEdit={false} (déjà supporté) et un nouveau prop showTallyForClosed={viewMode === 'owner' || isEnded} — pour que les résolutions clôturées affichent leur résumé tally.totalForQuotities / totalAgainstQuotities / totalAbstainQuotities en ligne dans la table.
  • Nouveau panneau « Mon vote » : carte qui liste les lots de cet utilisateur dans cette copro (lecture de myRepresentedLotsStore filtré sur coproprieteId === currentCoproprieteId) et, par lot, le statut de cast par résolution (Pour / Contre / Abstention / Pas encore voté). Pour une AG live, le panneau monte un CTA primaire « Rejoindre la session de vote » qui appelle getMyVoteLink.

Le syndic ne voit aucun changement — pour viewMode === 'syndic', tout est comme avant V1.1.5. C'est précisément l'intérêt de la bascule par rôle : zéro régression côté chef d'orchestre.

07 · La table des résolutions, vue copropriétaire

Ce que voit le copropriétaire d'une AG clôturée.

Aperçu inline du rendu — basé sur l'AG Brugmann 2024, avec trois résolutions représentatives : un vote ordinaire adopté, un point d'information, un vote en majorité renforcée rejeté.

# Question Type Majorité Clé Résultat
R1 Approbation des comptes 2023 Vote Absolue GEN ✅ Adoptée · 5 720 / 4 050 / 230
R2 Rapport du Conseil de Copropriété Information — — ℹ︎ Pour information
R3 Réfection de la toiture (~80 000 €) Vote 4/5 GEN ❌ Rejetée · 6 200 / 3 600 / 200

Le format du résultat, c'est pour / contre / abstention en quotités absolues — jamais en pourcentage, jamais en têtes. Un copropriétaire de Brugmann sait lire « 5 720 quotités pour » parce que c'est le format du PV qu'il reçoit chaque année. Cohérence entre dashboard et PV.

RGPD. La table ne rend pas les tableaux nominatifs namedAgainst / namedAbstain. Ces noms vivent dans le PV légal et y restent ; le dashboard donne les totaux, le PV donne les noms. C'est la lecture conservatrice du Code civil belge §3.87 §10 — le PV est l'artefact de transparence ; le dashboard est l'artefact opérationnel.

États avant / pendant le live. Pour les AG encore en convoked, la colonne « Résultat » lit « Pas encore ouverte » ou « À venir ». Pour une AG live avec une résolution courante en voting_open, la ligne montre un pill « En cours » avec un point pulsant — exactement le même feedback visuel que la vue chef d'orchestre, sans les contrôles d'ouverture / clôture.

08 · Rejoindre la session en direct

Le flux complet de getMyVoteLink.

Le bouton « Rejoindre la session de vote » du panneau Mon vote déclenche un appel callable, et seulement un. Voici le bout-en-bout, de la pression sur le bouton à l'ouverture de la page de vote dans un nouvel onglet.

  1. Frontend. Le bouton appelle httpsCallable('getMyVoteLink')({ coproprieteId, assemblyId }).
  2. Backend (functions/src/features/assemblies/api/getMyVoteLink.ts). Vérification d'auth : request.auth.uid doit figurer dans representedBy[] d'au moins un Owner doc de cette copro, ou être membre de l'org avec rôle ≥ 'admin'. Résolution de l'ownerId que l'utilisateur peut représenter — en V1 le lien lot-owner est 1:1, mais une représentation peut couvrir plusieurs Owner ; en V1.1.5 on émet un seul token frais par appel, scopé au premier Owner choisi.
  3. Génération du token. 32 octets d'aléa ; SHA-256 du secret ; révocation de tout AccessToken antérieur pour ce couple owner+assembly (revokedAt = serverTimestamp()) ; écriture d'un nouveau doc AccessToken indexé par le nouveau hash. Le tout dans une seule transaction Firestore.
  4. Retour. { url: 'https://<host>/vote/<secret>', expiresAt }.
  5. Frontend, suite. window.open(url, '_blank'). Le copropriétaire atterrit sur la page /vote/[token] existante, déjà gérée par les phases B + C — rien de neuf côté page de vote.

Trade-off à nommer. Le lien magique de l'e-mail continue de marcher jusqu'au premier clic du bouton dashboard ; à ce moment-là il est révoqué. C'est cohérent avec le substrat légal — un seul token actif par owner+AG — et ça donne au copropriétaire un chemin de réinitialisation propre s'il a perdu son e-mail. Côté dashboard, on ne stocke jamais le secret côté client : chaque clic régénère un secret frais, écrit le hash, et oublie le secret au prochain refresh de page.

La page /vote/[token] elle-même n'est pas modifiée par V1.1.5. Toute la logique de validation, de cast, de heartbeat est déjà en place depuis les phases B + C. V1.1.5 ne fait qu'amener le copropriétaire jusqu'à cette page depuis le dashboard, au lieu de l'obliger à rouvrir l'e-mail de convocation.

09 · Phasage

Cinq sous-phases, une à deux journées chacune.

V1.1.5 est un add-on sur V1.1, pas un module à part entière. Le phasage est calibré en jours, pas en semaines. Chaque sous-phase ferme proprement et peut être validée contre la recréation Brugmann avant de passer à la suivante.

F1

Cross-org store

myAssembliesStore.ts + fixtures de test unitaire. Vérification : un utilisateur de test avec des lots dans 2 copros voit les deux AG dans la sortie du store, triées correctement.

F2

Popover

Troisième section dans OrganizationSwitcher.svelte : pills de rôle + statut, tri, limite à 5 + voir tout. Vérification : snapshot test du dropdown pour le persona multi-copro Brugmann + Albert.

F3

Détail en lecture seule

Bascule viewMode dans [aId]/+page.svelte, prop showTallyForClosed sur TableAgenda, masquage de la table des livraisons. Vérification : un copropriétaire qui ouvre l'AG Brugmann 2024 voit ses 18 résolutions avec totaux, sans aucun bouton d'admin.

F4

Mon vote + getMyVoteLink

Cloud Function getMyVoteLink côté backend + panneau Mon vote côté frontend + bouton Rejoindre. Vérification : depuis le dashboard, le copropriétaire clique sur Rejoindre → /vote/[fresh-secret] s'ouvre dans un nouvel onglet et valide.

F5

i18n + polish + replay Brugmann

Strings FR + EN complets, parité dark/light, end-to-end : login en tant que copropriétaire Brugmann avec 1 lot, AG visible dans le popover, ouverture du détail, clic Rejoindre, vote, mise à jour du résultat affiché.

Une fois F5 livrée, V1.1.5 est terminée — environ une à deux semaines de travail concentré.

10 · Questions ouvertes

Huit points à trancher au fil des sous-phases.

Aucun ne bloque F1. Chacun se résout au moment où la sous-phase qui le rencontre démarre. La recommandation est ma meilleure lecture actuelle ; la décision finale revient à l'utilisateur.

# Question Recommandation
01 AG annulées dans le popover ? Non, masquées. Une AG annulée ne demande aucune action ; elle pollue la liste sans payer son ROI visuel.
02 Limite du popover : 5 / 10 / illimité ? 5 + lien « voir tout ». Couvre 95% des copropriétaires en multi-immeuble ; les autres cliquent.
03 Owner avec rôle admin sur la même copro : afficher l'AG une fois ou deux ? Une fois, badge Syndic (rôle dominant). Le syndic-aussi-copropriétaire fait son travail de syndic.
04 Bouton Rejoindre fait tourner le token — accepter de casser le lien e-mail à la première utilisation ? Oui. Un seul token actif par owner+AG, c'est le substrat légal. Le copropriétaire récupère un chemin propre via le dashboard.
05 Titulaire de procuration sans compte Syndicable : surface dans le dashboard ? Non en V1.1.5. Les procurations passent uniquement par e-mail ; le dashboard est réservé aux comptes authentifiés.
06 AG en cours, mais l'utilisateur n'a pas de lot concerné par la résolution courante : que voir ? Réutiliser l'UX « Cette résolution ne vous concerne pas » déjà spec'd au §7 de l'article V1.1. Pas de message neuf à inventer.
07 PV téléchargeable depuis le détail copropriétaire ? Oui, lien vers le doc historic_pv quand il existe. Pas de surface neuve, juste un bouton « Télécharger le PV » dans le panneau Mon vote.
08 Bascule owner ↔ syndic dans le popover : préserver le sélecteur d'org entre les rôles ? Oui, la route reste /organizations/[orgId]/.... Le rôle change, l'org ne change pas — c'est le sens même du viewMode.

12 · Colophon

Ce qu'est ce plan — et ce qu'il n'est pas.

C'est la conception de V1.1.5. Ce n'est pas un patch — aucun fichier de code n'est modifié par sa lecture. Ce n'est pas un engagement de livraison à N jours — le phasage suppose un seul implémenteur concentré, et chaque sous-phase peut respirer si Brugmann en révèle un cas non prévu.

Ce que cet article fige : la forme. Trois choix d'architecture, une liste de fichiers, une callable, cinq sous-phases. Le code commence à la validation. Si quelque chose dans la forme cloche — un nom de prop, un endroit dans la hiérarchie, une question ouverte mal résolue — c'est maintenant qu'il faut le dire, avant que les sous-phases ne s'enchaînent.

Ce qui n'est pas couvert : tout ce qui est dans la colonne OUT du §02. La route dédiée /my-assemblies, les notifications push, le téléchargement de PV in-app, la rédaction owner-side de résolution — autant de pistes pour V1.2 ou plus tard. V1.1.5 livre le minimum suffisant pour qu'un copropriétaire ait le sentiment d'être un utilisateur de premier ordre, sans que le périmètre déraille.
